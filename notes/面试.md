#### 量化gemm

+ 量化方法：

  + 对称量化[-127, 127]
  + **量化公式**: `quantized_value = round(original_value / scale)`
  + **反量化公式**: `dequantized_value = quantized_value * scale`

+ 量化粒度：

  **Per-Row量化** (A矩阵)：

  - **粒度**: 每行独立计算scale因子
  - **Scale计算**: `scale[i] = max(abs(row[i])) / 127`
  - **优势**: 每行数据分布独立优化，精度较高
  - **适用**: 激活矩阵（动态量化）

  **Per-Column量化** (B矩阵)：

  - **粒度**: 每列独立计算scale因子
  - **Scale计算**: `scale[j] = max(abs(col[j])) / 127`
  - **优势**: 权重矩阵预量化，推理时无额外开销
  - **适用**: 权重矩阵（静态量化）

+ **scale**：

  ```python
  # 存储的scale向量：
  A_scale: 形状为(M,)    # 只存储M个scale值，每行一个
  B_scale: 形状为(N,)    # 只存储N个scale值，每列一个
  
  # C矩阵的scale是通过广播计算的，不单独存储：
  C_scale[i,j] = A_scale[i] * B_scale[j]  # 运行时计算，不存储
  
  # 在kernel中的反量化代码：（kernel中传入A(int)、B(int)、C(fp)的指针）
  C_fp32 = accumulator.to(tl.float32) * A_scales[:, None] * B_scales[None, :]
  #        ^^^^^^^^^^^^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^
  #        int32矩阵乘法结果              (M,1)广播        (1,N)广播
  #                                    
  # 广播结果：(M,1) * (1,N) = (M,N) 的有效scale矩阵
  ```

+ 性能提升原因：

  1. **硬件Tensor Core优势** (~2-4x):
     - INT8 Tensor Core确实比FP16快2-4倍
  2. **内存带宽优势** (~1.5x):
     - 数据传输量减少33%
  3. **Cache效率提升** (~1.2-1.5x):
     - 更好的局部性，减少cache miss
  4. **PyTorch实现差异** (~1.2-1.8x):
     - Triton的专用优化 vs PyTorch的通用实现

#### flash attention

版本一：混合量化方案

- **量化范围**：仅Q和K使用INT8，V保持FP16
- **量化方式**：依赖硬件原生INT8矩阵乘法支持
- **关键代码**：

```python
qk = tl.dot(q, k).to(tl.float32)  # INT8矩阵乘法
qk = qk * q_scale[:, None] * k_scale  # 反量化到FP32
```

版本二：全量化方案

- **量化范围**：Q、K、V全部使用INT8
- **量化方式**：手动实现attention权重的量化
- **关键代码**：

```python
p = p * 127  # 手动量化attention权重
p = (p+0.5).to(tl.int8)  # 四舍五入量化
tmp = tmp * v_scale / 127  # 手动反量化
```

    q:(batch_size, num_heads, BLOCKSIZE_Q, head_dim)
    mi = -inf 	# (BLOCKSIZE_Q,)
    li = 1		# (BLOCKSIZE_Q,)  l = exp(QK^T-m)的每行之和
    acc = 0		# (BLOCKSIZE_Q, head_dim)
    for block_id in range(0, seq_len / BLOCKSIZE_KV):
        S = QK^T * sqrt(head_dim)
        mi_new = rowmax(S)
        P = exp(S-mi)

        alpha = exp(mi-mi_new)
        acc += PV * alpha
    acc /= li[;, None]		# 除分母完成softmax

+ 外层循环Q为什么性能更好？
  + 减少内存访问：Q放在外层循环是“一对多”, 加载一块Q后一直保存在寄存器中重复使用；
    KV放在外层循环是“多对一”，每个内层循环需要多次重复读取Q块

+ online softmax
  + 维护QK^T的每行最大值
  + 修正系数=exp(上次最大值-当前最大值)

#### 性能分析脚本

**概述**
这个脚本主要就是通过Python装饰器，把原始函数包装一下，再函数调用前后用time方法记一下时间，最后就可以统计推理过程所有函数的调用次数和执行时间。它里面是不断递归地装饰背后整个模块的所有函数，包括整个类、forward函数、一直到最底层的加减乘除算子调用。最后测出来的结果跟profiler工具是差不多的，也能用在SAS模型上，虽然还是测不了具体每个算子的调用耗时，但可以测出来模型的每个层的推理耗时。



**介绍一下Python装饰器**
	装饰器是python的一个内置功能，就是它接受一个函数作为参数，返回一个新的函数。装饰器优点就是可以在不修改原函数代码的情况下，给函数添加额外功能。
	装饰器的用途非常广泛：日志记录（记录函数调用和参数）、权限验证（检查用户是否有权限执行某操作）、缓存机制（避免重复计算）、重试机制（函数失败时自动重试）、参数验证（检查输入参数是否合法）、异常处理等。



**这种监控方式对性能的影响如何？**
 确实会有一定性能开销，主要来自于时间记录和字典操作。不过对于性能分析场景，这种开销其实无所谓，因为重点是用同一个测试方法来对比两边算子的性能嘛，关注的是相对性能差异。所以大家都有这些性能开销。





### 推理框架项目

#### 推理流程

**宏观上**，其实就是"用户输入prompt，模型给你回答"。那从微观上来看就有很多步嘛，

**第一步是请求预处理**。用户发来一句话，比如"帮我写个Python函数"，我们首先要把它tokenize，就是切成一个个数字ID。这里有个细节，如果是对话模型，还要加上对话模板，比如把用户输入包装成特定格式。

**然后进入模型推理阶段**，这是最核心的部分。模型会逐层计算，每一层都有attention和MLP两个主要组件。这里有个关键优化叫KV Cache - 就是把之前计算过的key和value缓存起来，避免重复计算。不然每生成一个token都要重新算一遍所有历史token，那计算量就爆炸了。

**接下来是文本生成**，也就是decoding。模型输出的是每个词的概率分布，我们要从中选择下一个token。可以用贪心搜索直接选概率最高的，也可以用采样方法增加多样性。这个过程会一直重复，直到生成结束符或者达到最大长度。

**从工程角度看**，真正的挑战在于如何高效处理多个并发请求。传统做法是static batching，就是等一批请求都处理完再处理下一批。但这样效率很低，因为不同请求的生成长度不一样。

**现在主流的做法是continuous batching**，比如vLLM就用这个技术。简单说就是，当batch里有请求生成完了，立马把它移除，然后从队列里拿新请求填进来。这样GPU利用率能保持很高。

**内存管理也是个大问题**。KV Cache会随着序列长度线性增长，很容易OOM。vLLM提出了PagedAttention，把KV Cache像操作系统管理内存一样分页存储，大大提高了内存利用率。我之前调试过一个7B模型的服务，用了PagedAttention后，同样显存能处理的并发数提升了好几倍。

**在实际部署中**，还有很多细节需要考虑。比如动态batch size调整 - 根据当前GPU内存使用情况自动调整批次大小。还有就是模型并行，对于很大的模型，需要把它切分到多张卡上。

**性能监控也很重要**。我们通常关注几个指标：吞吐量（tokens/second）、延迟、GPU利用率。在线服务的话，还要监控队列长度，避免请求堆积。

我觉得做LLM推理优化，核心思路就是最大化GPU利用率。GPU这种昂贵资源，每一秒的空闲都是浪费。所以无论是continuous batching、KV Cache、内存分页，本质上都是为了让GPU始终保持高负载运行。

当然，不同场景的优化策略也不一样。如果是交互式对话，更关注首token延迟；如果是批量处理，更关注总吞吐量。这需要根据具体业务需求来调优。一次推理流程

### Triton基础概念

#### 1. triton编译流程

> triton kernel -> triton IR -> LLVM IR -> PTX，最后配合 runtime 运行

上述编译过程通过 `@triton.jit` 装饰器完成，具体来说是遍历提供的 Python 函数的抽象语法树（AST），并使用常见的 `SSA` 构建算法即时生成 `Triton-IR`。然后，编译器后端会简化、优化并自动并行化所产生的 `IR` 代码，再将其转换为高质量的 `LLVM-IR`，最后生成 PTX 并在 NVIDIA GPU 上执行。

#### 3. num_warps和num_stages

num_warps —— **空间并行(同时处理更多数据)**

+ 用于指定一个**block的线程数**, 即**每个block的线程数为num_warps * 32**
+ 1 warp = 32 threads (硬件写死), 这32个线程**严格并行**，同时执行相同的指令
+ block内部并不是所有线程严格并行，而是**以warp为原子单位并行+warp层面的调度切换**
+ num_warps越大，kernel并行度越高；但所有warp会**共享所在SM的寄存器和共享内存**，因此**num_warps设置过大会导致每个线程可用的共享内存或寄存器变少**



num_stages —— **时间并行(流水线重叠)**

+ 表示**流水线深度**, 表示可以同时进行多少阶段的数据加载/计算
+ 过多的流水线阶段虽然有助于延迟隐藏，但同时会增加寄存器压力和同步开销。



#### kernel性能分析

profiler看指标：GPU利用率、显存占用/L2 cache占用

1. 显存占用低，GPU利用率低(或高): **compute bound(计算快、数据读不过来)**，并行度不够->
   + 减小BLOCKSISZE, 增大block数，充分利用SM(一个block对应一个SM)
   + 增大num_warps, 即增大并行线程数
2. GPU利用率低，显存占用高：**memory bound（GPU闲、显存忙）**, 
   + 增大num_stages:减少内存延迟
   + 增大矩阵乘法BK：减少显存读取次数，增加数据复用次数

### GPU 并行计算基础

- **CUDA 的线程模型：warp、thread、block 有何区别？**
   **答：** 在 CUDA 中，每个多处理器（SM）同时调度执行一个 warp 的指令，当前架构下一个 warp 包含 32 个线程。线程（thread）是最小执行单元，多个线程组成线程块（block），多个线程块组成网格（grid）。程序中通过配置每个 kernel 的 block 大小（每个 block 包含多少线程）和 grid 大小来控制并行度。一个线程块内的线程共享同一块 SRAM（共享内存），线程块之间可以看作无直接同步（除非使用全局同步原语）。概括：每个 block 有若干 warp，warp 内的 32 个线程同时执行同一条指令；多个 block 在多个 SM 上并发执行。
- **什么是内存合并 (coalescing)？如何避免冲突？**
   **答：** 内存合并指的是当同一个 warp 的线程同时访问相邻地址时，硬件会将多个内存访问合并为少量传输。例如在 Volta 及以后架构上，一个 warp 可以以 128 字节（32×4-byte）的块为单位访问，只要每个线程访问连续地址且对齐，就能打包为一次传输。反之，如果访问不对齐或散乱，就可能产生多个事务，降低带宽利用率。在 Triton 中，使用 `tl.arange` 等生成连续地址序列可以自动获得合并访问。另外，共享内存中也要避免**bank 冲突**，即同一时刻过多线程访问同一个 bank（16/32 位交错的片段）；编写代码时可通过添加填充或调整访问模式来消除冲突。
- **什么是占用率 (Occupancy)？为什么重要？**
   **答：** 占用率定义为每个 SM 上**活跃 warp 数**与该 SM 最大活跃 warp 数的比值。换言之，若 SM 最多可并行 64 个 warp，当活跃的 warp 只有 32 个时，占用率为 50%。高占用率意味着有更多并发线程可执行，可以更好地隐藏指令和内存的延迟；但过高占用率也可能导致每线程资源减少（如寄存器变少，产生 spill），从而降低性能。通常用 CUDA 提供的 Occupancy API（如 `cudaOccupancyMaxActiveBlocksPerMultiprocessor`）或 Nsight 工具来评估占用率。在调优时，若发现占用率过低，可增大 block 大小或减少每线程使用的资源；若占用率已满，还需关注是否有其他瓶颈（如内存带宽）。

### 推理加速

#### 1. 数据并行DP/流水线并行PP/张量并行TP/专家并行EP

| 维度         | DP (数据并行)                                 | PP (流水线并行)                        | TP (张量并行)                                             | EP (专家并行)                            |
| ------------ | --------------------------------------------- | -------------------------------------- | --------------------------------------------------------- | ---------------------------------------- |
| **基本原理** | 每个设备保存完整模型副本，处理不同的数据batch | 模型按层间切分，数据在设备间流水线传递 | 模型按层内划分（例如attention多头），在多个设备上并行计算 | 不同专家模块分布在不同设备，动态路由选择 |
| **切分维度** | 数据维度                                      | 模型层维度                             | 权重维度                                                  | 专家维度                                 |
| **通信模式** | All-Reduce同步梯度                            | 点对点传递激活值                       | 频繁All-Reduce/All-Gather                                 | 动态All-to-All通信                       |
| **延迟**     | 几乎没有                                      | 中                                     | 高                                                        | 中                                       |
| **通信频率** | 每个step结束时                                | 每层计算完成时                         | 每个算子内部                                              | 每个token的专家选择时                    |
| **内存占用** | 高（每设备存完整模型）                        | 中（按层数均分）                       | 低（按并行度均分）                                        | 中（取决于专家分布）                     |
| **计算效率** | 高（无额外开销）                              | 中（存在bubble time）                  | 高（细粒度并行）                                          | 高（专家稀疏激活）                       |
| **适用场景** | 小模型，多数据                                | 大模型，层数多                         | 超大模型，单层很大                                        | MoE模型                                  |
| **扩展性**   | 受模型大小限制                                | 受模型深度限制                         | 受单层大小限制                                            | 受专家数量限制                           |
| **负载均衡** | 天然均衡                                      | 可能不均衡                             | 天然均衡                                                  | 需要动态调整                             |
| **容错性**   | 好（任一设备故障影响小）                      | 差（单点故障影响大）                   | 差（紧密耦合）                                            | 中（专家可重分配）                       |

**Tensor Parallel（TP）**：把模型按层内切分，比如attention的多头、MLP的参数都分布到不同GPU上，需要频繁的all-reduce通信。

**Pipeline Parallel（PP）**：按层间切分，不同GPU负责不同的layer，通信相对较少，但可能有bubble。

**Data Parallel**：每个GPU都有完整模型，处理不同的batch，适合小模型。

**Expert Parallel**：把不同的expert分布到不同的GPU上。

+ 首先router决定每个token要路由到哪些expert，然后做一个all-to-all通信，把token发送到对应的expert所在的GPU。
+ expert计算完后，再做一次all-to-all把结果发回去。



##### DP (数据并行)

**优点：**

- 实现简单，框架支持好
- 完美负载均衡
- 容错性强

**缺点：**

- 模型必须能放入单个设备
- 内存利用率低
- 通信量随模型大小线性增长

**典型应用：** 中小型模型的训练和推理



##### PP (流水线并行)

**优点：**

- 能训练超大模型
- 通信量相对较少
- 实现相对简单

**缺点：**

- 存在bubble time降低效率
- 需要micro-batch技术优化
- 负载可能不均衡

**典型应用：** GPT系列大模型训练



##### TP (张量并行)

**优点：**

- 内存效率最高
- 能处理超大单层模型
- 计算高度并行

**缺点：**

- 通信频繁，对网络要求高
- 实现复杂度高
- 扩展性受限于单层大小

**典型应用：** Transformer的attention和MLP层



##### EP (专家并行)

**优点：**

- 模型容量大但计算稀疏
- 可以不均匀分布专家
- 支持动态负载均衡

**缺点：**

- 只适用于MoE架构
- 负载均衡复杂
- 通信模式不规则

**典型应用：** Switch Transformer、GLaM等MoE模型



#### 2. MHA->MLA

![image-20250824153702982](./../img/typora-user-images/image-20250824153702982.png)

MQA、GQA、MLA都是为了压缩KV cache嘛，MQA是所有head都共享一个KV，虽然内存压缩的特别多，但是性能也会明显下降。所以GQA就是一个改进，n个head分成g个组，每个组的共享一个KV，就属于一种折中的办法，介于MHA和MQA之间。


MLA的话我觉得算是一个新的算法了，跟MQA、GQA直接减少KV不一样。

首先因为它存的不是KV，而是叫compressed latent KV。具体来说就是把原来计算KV的过程拆成了两步，第一步计算出矩阵c（就是compressed矩阵），第二步再通过C乘上两个权重分别得到K和V。这个思想就类似低秩分解，最后只需要存这个c矩阵，不需要存K和V，推理的过程中把KV算出来，并且KV的数量是和原来MHA是一样，所以说效果完全没有下降。

不过这样的话计算量就上升了，因为推理的过程得重新计算KV嘛，所以这和不用KV cache有什么区别呢？MLA的解法就是叫矩阵吸收，就是从C计算得到K和V的那两个矩阵是不需要存的，C到K的那个矩阵可以跟Wq合并，也就是说输入x乘上合并后的矩阵直接就可以得到$QK^T$，这样的话就不需要先算出Q和K再做矩阵相乘了；还有C到V的那个矩阵可以和WO合并，这样的话就不需要先计算出V，再和softmax($QK^T$)做矩阵相乘了，直接乘上合并后的矩阵就可以得到最后的结果O.

所以说MLA它整个计算过程已经完全变样了，我感觉都可以叫一个新的算法了。



**为什么MHA计算时要除$\sqrt{h}$（h=head_dim)?**

+ 为了维持方差稳定，防止梯度消失
+ 如果不除，$QK^T$方差会特别大，导致softmax之后只有0和1了，注意力分布就成了one-hot
+ 反向传播时，softmax的导数在|x|比较大时接近0，这样就会梯度消失




#### 3. Paged Attention

> KV Cache 被划分为块。块在内存空间中不需要连续。

**Paged Attention原理：**

传统LLM预分配KV cache都是按照最大长度max_seq_len来分配，这样的话就很容易浪费，因为不同序列长度不一样，如果有些用户输入的prompt很短就结束对话了，那给他分配的这么多内存没用到的都浪费了。

Paged Attention的优化思想就是把KV cache的分配以块为单位，比如说16个token算一个page。这样的话就可以很明显地减少内存浪费问题，只有最后一页会存在浪费，vllm说是可以把内存利用率从20%以下提升到90%几。
并且每个page在内存中是不需要连续的，因为有单独的一个页表，每个元素代表这一页在内存中的地址。只要逻辑上连续就行。
还有一个优点就是可以内存共享，几个sequence可以共享同一个page。比如说模型一开始的系统prompt就可以被所有用户共享；不同用户对话过程中遇到连续的相同token也可以共享那个page。



**如何判断几个sequence可以共享一个page？**

建Prefix Tree，每来一个新序列，就沿着树一个token一个token的走一遍，如果走过了一个page的数量，就说明这个page已经存在了，共享存在的那个page就行。
虽然还是一个token一个token的比较，但是存储降低了很多，不需要每个sequence都存下来一个一个对比。

#### 4. continuous batching

> **核心：动态管理KV cache和batch**

**1. KV Cache管理**

每个请求有自己的KV cache（之前生成的key-value）。传统方法是请求结束才释放，continuous batching中请求完成立即释放，为新请求腾空间。

**2. 请求队列**

维护等待队列和运行队列。新请求入队，根据显存和batch大小动态加入运行batch，完成请求移出。

**3. 动态batch组织**

每次推理不是固定batch大小，而是根据：

- 当前运行请求数
- 剩余显存
- KV cache占用

动态决定这一轮推理的batch大小。

**4. Padding和Attention Mask**

由于请求长度不同，需要：

- Padding统一长度
- Mask标记有效token位置

完成的请求token全标0，新请求补充上来时重新计算mask。

**5. 调度策略**

- FCFS：先来先服务
- Priority：高优先级请求先完成
- 最长优先/最短优先

**关键难点：** 频繁加入/移出请求，KV cache分散，需要内存管理器追踪每个请求的cache位置





#### 5. Per-channel vs Per-group 

Per channel是每个output channel用一个scale，粒度更细，量化精度更高。Per group是把多个channel分成一组，一组用一个scale。

虽然per channel精度更高，但是dequantization的时候计算开销更大，因为要对每个channel做不同的scale操作。Per group的话，可以用SIMD指令batch处理，速度更快。

而且现在的量化算法，比如AWQ、GPTQ，在per group的粒度下已经能取得很好的精度，所以工程上更倾向于用per group，在精度和速度之间找平衡。



#### 6. RMSNorm vs LayerNorm

##### 基本概念

+ 这两个都是归一化技术，在Transformer里面起到稳定训练、加速收敛的作用。
+ 让数据分布更稳定，避免出现数值爆炸或者消失的问题。
+ 在模型中的位置: 每个Transformer block的attention前面和MLP前面

##### RMSNorm优势

1. 计算量少了25%左右，因为**不用算均值和方差**了
2. 数值稳定性反而更好，因为避免了减均值的精度损失
3. 内存占用更少，中间变量更简单
4. 在GPU上更好优化，一次遍历就搞定

##### 公式

**给定输入向量** $x \in \mathbb{R}^d$,

**LayerNorm:**

1. 算均值$$\mu = \frac{1}{d} \sum_{i=1}^{d} x_i$$
2. 算方差$$\sigma^2 = \frac{1}{d} \sum_{i=1}^{d} (x_i - \mu)^2$$
3. 归一化+仿射变换$$\text{LayerNorm}(x) = \gamma \odot \frac{x - \mu}{\sqrt{\sigma^2 + \varepsilon}} + \beta$$, 其中:
   + $\gamma \in \mathbb{R}^d$：可学习的缩放参数向量
   + $\beta \in \mathbb{R}^d$：可学习的偏移参数向量  
   + $\varepsilon$：数值稳定性常数（通常为 $10^{-6}$）

**RMSNorm**:

1. 算均方根$$\text{RMS}(x) = \sqrt{\frac{1}{d} \sum_{i=1}^{d} x_i^2}$$
2. 归一化+缩放$$\text{RMSNorm}(x) = \gamma \odot \frac{x}{\text{RMS}(x) + \varepsilon}$$, 其中:
   + $\gamma \in \mathbb{R}^d$：可学习的缩放参数向量
   + $\varepsilon$：数值稳定性常数（通常为 $10^{-8}$，比LayerNorm更小）
   + 注意：**没有偏移量** $\beta$

| 归一化方法 | 参数            | 计算复杂度 | 公式特点                     |
| ---------- | --------------- | ---------- | ---------------------------- |
| LayerNorm  | $\gamma和\beta$ | O(2d)      | 减均值 + 除标准差 + 仿射变换 |
| RMSNorm    | 只有$\gamma$    | O(d)       | 除均方根 + 缩放              |







### LLM基础

#### 1.LayerNorm & BatchNorm

`Batch Norm` 和 `Layer Norm` 的区别一句话总结就是 `bn` 是切特征，`ln` 是切样本。

+ `BN`: 对于每个特征维度，计算它在**整个批次中**的均值和标准差，然后对该特征进行归一化。
+ `LN`: **对每个样本**单独计算其所有特征的均值和标准差，然后在该样本内进行归一化。



#### 2. LLM两个推理阶段

Prefill 和 Decode

+ Prefill: 并行计算所有prompt token的QKV，生成第一个输出token。然后算出来的KV就存到KVcache里
+ Decode: 每生成一个token，计算这个token的QKV，把KV拼接到所有token的KV矩阵上，接着一行Q和整个KV矩阵进行计算，生成下一个token，重复这个过程直到结束。

为什么Prefill阶段要计算所有token的Q？明明生成一个token只需要最后一行Q即可，prefill阶段不能只计算最后一行Q吗？

+ 本质原因就是一个大模型有很多和attention layer，每个attention的输入是上一层的输出，并且有各自的KV cache。
  如果第一个attention layer只计算最后一行Q，计算得到的O也只有一行，那下一层attention就只能得到一个token作为输入了，这样的话后面所有attention layer就完全没有历史信息了，也没有KVcache了，这些attention layer就废了。
+ 所以prefill计算所有token的Q就是为了让全部的attention layer把自己的KVcache计算出来。

#### 4.Temperature/top-P/top-K

 <img src="./../img/typora-user-images/image-20250824143904935.png" alt="image-20250824143904935" style="zoom:33%;" />

**temperature**通过对logits的值进行缩放（除以T)，从而修改softmax后的概率分布

 <img src="./../img/typora-user-images/image-20250824144126434.png" alt="image-20250824144126434" style="zoom: 50%;" /> <img src="./../img/typora-user-images/image-20250824144148185.png" alt="image-20250824144148185" style="zoom:50%;" />

+ T取值一般在0.0~2.0, 最常用范围是0.2~1.0
+ $T\in(0.2, 0.6)$: 低温度，logits放缩后绝对值会变大，softmax后又会放大logits值之间的差距，
  从而**高概率token的概率会变得更高，低概率token的概率会变得更低**，概率分布更集中->模型输出更准确、减少随机性
+ $T\in(0.7, 1.0)$: 中等温度，logits放缩后绝对值会变小，softmax后会压缩logits值之间的差距，
  从而**高概率token的概率会降低，低概率token的概率会提高**，概率分布更平坦->增加模型创造性**（GPT默认0.7）**
+ $T > 1$: 高温度，容易产生幻觉。适用于艺术性生成场景。



**top-P/top-K**用与选择输出token的范围, 概率太低的直接不考虑。（**不是每次都在整个vocab size中选token输出！）**

 <img src="./../img/typora-user-images/image-20250824145210948.png" alt="image-20250824145210948" style="zoom:33%;" />

+ top-K指定token数量: 直接选择概率最高的K的token
+ top-P指定token总和: 选择概率总和>=P的最少token（即从高到低开始选，总和达到P时停止）

**Top-P** 通常比 **Top-K** 更受欢迎，因为它能根据概率分布的实际情况动态调整考虑的token数量，使其更灵活。而 Top-K 总是考虑固定数量的选项，这可能在概率急剧下降时过于宽泛，或者在概率分布分散时过于狭窄





#### 4.RAG vs Fine-tuning

我感觉这俩好像不是一个维度的...

SFT就是拿另一个比如说某个领域的数据集，把这个模型继续训练一下，那训练的时候肯定**会修改模型参数**嘛，而且比较**耗资源**。

RAG是在跟模型对话的时候起作用的，就是他会去查一个外部的知识库，然后把查到的内容当成prompt跟你的输入一起丢给大模型，最后得到输出。那这个肯定**不会修改模型参数**的。

而且RAG和SFT是可以并存的呀，微调之后还是可以RAG查外部知识库的。所以RAG优点就是可以**获取最新的知识**，也不需要训练模型，并且基本**不会有幻觉**的风险，因为背后有资料嘛。SFT的话就会把模型整个能力、风格都改掉，一般都是企业要搞一个领域大模型才会话很多资源去微调。

##### `=delete`和`=default`函数区别

+ `=delete`表示该函数不能被调用。
+ `=default`表示编译器生成默认的函数

**`= default` 的使用范围：**
 只能用于编译器可以自动生成的特殊成员函数，包括：

- 默认构造函数
- 拷贝构造函数
- 移动构造函数
- 析构函数
- 拷贝赋值运算符
- 移动赋值运算符

```cpp
class MyClass {
public:
    MyClass() = default;                    // 默认构造函数
    MyClass(const MyClass&) = default;     // 拷贝构造函数
    MyClass(MyClass&&) = default;          // 移动构造函数
    ~MyClass() = default;                  // 析构函数
    MyClass& operator=(const MyClass&) = default;  // 拷贝赋值
    MyClass& operator=(MyClass&&) = default;       // 移动赋值
};
```

**`= delete` 的使用范围：**
 可以用于任何函数，用途更广泛：

```cpp
class MyClass {
public:
    // 禁用特殊成员函数
    MyClass(const MyClass&) = delete;           // 禁用拷贝构造
    MyClass& operator=(const MyClass&) = delete; // 禁用拷贝赋值
    
    // 禁用普通成员函数
    void someFunction() = delete;
    
    // 禁用特定参数类型的重载
    void process(int value);
    void process(double value) = delete;  // 禁用double版本
};

// 全局函数也可以使用delete
void globalFunction(int) = delete;
```

`= delete` 特别有用的场景是防止不想要的隐式类型转换，或者明确禁用某些函数重载，这样的设计比简单的私有化更清晰明确。



##### `std::move`和右值引用

**左值和右值：表达式的“值类别”**

+ **左值：** 有身份，有内存地址，生命周期长。例：变量x、"abc"、*p等
+ **右值：** 临时对象，即将销毁。例：42、a+b等

**左值引用和右值引用：引用类型**

```c++
int x = 5;      // x是左值(值类别)  5是右值（临时值）
int& lref = x;   // lref是左值引用类型, lref本身是左值(值类别) 
int&& rref = 5;    // rref是右值引用类型, 但rref本身是左值(值类别)
```

+ `std::move`作用是把左值转换为**右值引用（不是右值）**，返回值类型为右值引用(`T&&`)
+ move本身不创建新对象，只是标记"这个对象可以被窃取"

```c++
string s1 = "hello";
string s2 = s1;              // 调用拷贝构造（深拷贝）
string s3 = std::move(s1);   // 调用移动构造

cout << s1 << endl;    // 输出：""（空字符串，内容被移动了）
cout << s2 << endl;   // 输出："Hello"
cout << s3 << endl;   // 输出："Hello"
```



##### 智能指针

常用三种:`unique_ptr`, `shared_ptr`, `weak_ptr`

1. `unique_ptr`:独占所有权。只能有一个unique_ptr指向同一个对象或数组，超出作用域自动删除。

   ```c++
   std::unique_ptr<int> ptr = std::make_unique<int>(42);
   // 自动管理内存，作用域结束时自动删除
   // 不能拷贝，只能移动
   std::unique_ptr<int> ptr2 = std::move(ptr);  // 转移所有权
   // 现在ptr是nullptr，ptr2拥有内存
   ```

2. `shared_ptr`:共享所有权。允许多个shared_ptr指向同一个对象，通过引用计数来管理资源的生命周期。只有当最后一个shared_ptr超出作用域或被显式释放时，资源才会被删除。

   ```c++
   std::shared_ptr<int> ptr1 = std::make_shared<int>(42);
   std::shared_ptr<int> ptr2 = ptr1;  // 可以拷贝，引用计数+1
   
   std::cout << ptr1.use_count() << std::endl;  // 输出：2
   // 当最后一个shared_ptr销毁时，才会删除内存
   ```

3. `weak_ptr`:弱引用。它可以指向由shared_ptr管理的对象，但不会增加引用计数。主要用于解决shared_ptr的循环引用问题

   ```c++
   std::shared_ptr<int> sptr = std::make_shared<int>(42);
   std::weak_ptr<int> wptr = sptr;  // 不影响引用计数
   
   if (auto locked = wptr.lock()) {  // 检查对象是否还存在
       std::cout << *locked << std::endl;  // 安全访问
   }
   // 主要用于打破循环引用
   ```

#### C++内存管理

##### 内存分区

 <img src="./../img/typora-user-images/image-20251019134527883.png" alt="image-20251019134527883" style="zoom:33%;" />

+ **栈**: 函数中的局部变量、形参、返回值。编译器自动管理
+ **堆**: **内存区域不连续！！！存储结构为链表**。通过new/malloc动态分配的内存，由程序员手动分配释放
+ 全局区/静态区（**static**): 
  细分为
  1. 已初始化的数据段(.data): 存放已初始化的全局/静态变量
  2. 未初始化的数据段(.bss): 存放未初始化（**或初始化为0**）的全局/静态变量
+ 常量区（.rodata): 是静态区的一个段，**属于静态区！**存放字符串常量、const全局变量、其他**只读数据**

```C++
const static int a = 10;		// 常量区(.rodata)
const int b = 10;				// 常量区(.rodata)
static int c = 10;				// 全局/静态区(.data)——已初始化
int main() {
    const static int d = 10;	// 常量区(.rodata)
    const int e = 10;			// 栈
    static int f = 10;			// 全局/静态区(.data)——已初始化
}
```

1. **static决定在栈还是静态区，变量加上static相当于变成了全局变量**
   + 局部静态：只改变生命周期，不改变作用域（还是局部）
   + 全局静态：不改变生命周期，但会**限制**作用域（只允许本文件访问）

2. const只表示**只读**，不改变存储区，**只改变存储段**  ——**const变量必须初始化！！！**
   + 对于全局/静态变量，const会把全局区移动到常量区



##### 内存对齐

**概念：**
 编译器会让变量的地址是其大小的整数倍（或某个规则的倍数）。

**原因：**

1. **硬件要求：** 有些CPU访问未对齐的数据会出错或效率低
2. **性能：** 对齐的数据访问更快，可能一次内存访问就能读取

**规则：**

- char：1字节对齐
- short：2字节对齐
- int：4字节对齐
- double：8字节对齐
- 结构体：按最大成员对齐

```C++
struct s1 {
    char c1;	// 1字节
    char c2;	// 1字节
    double d;	// 8字节
};				// 8(1+1+内存对齐6) + 8 = 16字节
struct s2 {
    char c1;	// 1字节
    double d;	// 8字节
    char c2;	// 1字节
};				// 8(1+内存对齐7) + 8 + 8(1+内存对齐7) = 24字节
```

##### `static`

> **核心：一个静态成员只有一个实例**

1. 静态成员变量：属于类本身而非某个具体对象，被所有对象共享

   + 存储在全局（静态）区

   + 必须在类外定义+初始化，类内只能声明
     ```c++
     class Student {
         static int count;  // 声明
     };
     int Student::count = 0;  // 类外定义初始化
     ```

   + 可通过类名直接访问

   + 没有创建对象时，静态成员变量也存在

2. 静态成员函数：属于类本身而非对象，**没有this指针**

   + 只能访问静态成员变量及其他静态成员函数
   + **不能访问非静态成员**（因为没有this）
   + 不能被声明为`const`或`virtual`
   + 可通过类名直接调用

##### 多态

> 定义：同一个操作作用于不同的对象时，可以产生不同的行为 / 不同的对象对同一消息（函数调用）做出不同的响应
> **主要通过`virtual`虚函数和继承实现**: `virtual`标志这个函数可以被派生类重写

分类：

+ **编译时多态（静态多态）**：通过函数重载和运算符重载实现，在**编译时**确定调用哪个函数
+ **运行时多态（动态多态）**：通过虚函数实现，在**运行时**根据对象的实际类型决定调用的函数。



**虚函数实现机制**

虚函数通过虚函数表来实现。虚函数的地址保存在虚函数表中，在类的对象所在的内存空间中，保存了指向虚函数表的指针（称为“虚表指针”），通过虚表指针可以找到类对应的虚函数表。虚函数表解决了基类和派生类的继承问题和类中成员函数的覆盖问题，当用基类的指针来操作一个派生类的时候，这张虚函数表就指明了实际应该调用的函数。

+ 虚函数表**存放的内容**：类的虚函数的地址。
+ 虚函数表**建立的时间**：编译阶段，即程序的编译过程中会将虚函数的地址放在虚函数表中。
+ **虚表指针保存的位置**：虚表指针存放在对象的内存空间中最前面的位置，这是为了保证正确取到虚函数的偏移量。

##### 虚函数&纯虚函数

虚函数: 被`virtual` 关键字修饰的成员函数，就是虚函数。

纯虚函数: 在基类中声明的虚函数，它在基类中没有定义，任何派生类都必须定义自己的实现。

```c++
virtual void function() = 0;	// =0表示纯虚函数
```

+ 含有纯虚函数的类被称为抽象类。
+ 抽象类不能被实例化，只能作为接口使用。
+ 派生类必须实现所有的纯虚函数，否则该派生类也会变成抽象类。



##### 为什么析构函数(基类)一般是虚函数？

> 主要是为了解决**基类指针指向派生类对象时**的资源释放问题

+ 如果我们有一个基类指针，它实际上指向一个派生类对象，当我们删除这个基类指针时，如果析构函数不是虚函数，那么就**只会调用基类的析构函数，而不会调用派生类的析构函数**。这可能会导致派生类对象的一些资源没有被正确释放，从而引发内存泄漏等问题。
+ 如果我们将析构函数设置为虚函数，那么在删除基类指针时，会**首先调用派生类的析构函数，然后再调用基类的析构函数**，从而确保所有的资源都能被正确释放。

##### 为什么构造函数(所有类)不写成虚函数？

+ 这是 C++ 语言的根本规则。虚函数机制依赖于对象创建完成后才存在的虚函数表（vtable），而构造函数本身就是创建对象的入口。在对象尚未完全创建好之前，虚函数机制无法工作。

##### 几个特殊成员函数（缺省函数）

> **构造函数**：创建新对象时用
>
> **赋值运算符**：给已存在的对象重新赋值时用
>
> **拷贝**：完整复制另一个对象的内容
>
> **移动**：从临时对象"偷取"资源，效率更高
>
> **析构函数**：对象消失时自动清理资源

1. 拷贝构造函数

   **作用：** 用另一个同类型对象来创建新对象时调用

   ```c++
   class Person {
   public:
       string name;
       int age;
       
       // 拷贝构造函数
       Person(const Person& other) {
           name = other.name;
           age = other.age;
           cout << "拷贝构造函数被调用" << endl;
       }
   };
   
   Person p1("张三", 25);
   Person p2 = p1;        // 这里会调用拷贝构造函数
   Person p3(p1);         // 这里也会调用拷贝构造函数
   ```

2. 移动构造函数
   **作用：** 从临时对象"偷取"资源来创建新对象，避免不必要的拷贝

   ```C++
   class Person {
   public:
       string name;
       
       // 移动构造函数
       Person(Person&& other) {
           name = move(other.name);  // "偷取"而不是拷贝
           cout << "移动构造函数被调用" << endl;
       }
   };
   
   Person p1 = Person("李四", 30);  // 临时对象会触发移动构造
   ```

   

3. 析构函数
   **作用：** 对象被销毁时自动调用，用于清理资源

   ```c++
   class Person {
   private:
       int* data;
   public:
       Person() { data = new int(42); }
       
       // 析构函数
       ~Person() {
           delete data;  // 释放内存
           cout << "析构函数被调用，清理资源" << endl;
       }
   };
   
   {
       Person p;  // 创建对象
   }  // 离开作用域时，自动调用析构函数
   ```

4. 拷贝赋值运算符
   **作用：** 把一个已存在对象的值赋给另一个已存在的对象

   ```C++
   class Person {
   public:
       string name;
       
       // 拷贝赋值运算符
       Person& operator=(const Person& other) {
           if (this != &other) {  // 防止自赋值
               name = other.name;
           }
           cout << "拷贝赋值运算符被调用" << endl;
           return *this;
       }
   };
   
   Person p1("王五", 20);
   Person p2("赵六", 25);
   p2 = p1;  // 这里调用拷贝赋值运算符，p2的内容被p1覆盖
   ```

5. 移动赋值运算符
   **作用：** 从临时对象"偷取"资源赋值给已存在的对象

   ```C++
   class Person {
   public:
       string name;
       
       // 移动赋值运算符
       Person& operator=(Person&& other) {
           if (this != &other) {
               name = move(other.name);  // "偷取"资源
           }
           cout << "移动赋值运算符被调用" << endl;
           return *this;
       }
   };
   
   Person p1("孙七", 35);
   p1 = Person("周八", 40);  // 临时对象触发移动赋值
   ```

#### 进程与线程

 <img src="./../img/typora-user-images/image-20251020133048530.png" alt="image-20251020133048530" style="zoom:33%;" />

- **本质区别**：进程是操作系统资源分配的基本单位，而线程是任务调度和执行的基本单位
  + 资源指的是**虚拟内存、文件句柄、信号量**等资源。
- **在开销方面**：每个进程都有独立的代码和数据空间（程序上下文），程序之间的切换会有较大的开销；线程可以看做轻量级的进程，同一类线程共享代码和数据空间，每个线程都有自己独立的运行栈和程序计数器（PC），线程之间切换的开销小
- **稳定性方面**：进程中某个线程如果崩溃了，可能会导致整个进程都崩溃。而进程中的子进程崩溃，并不会影响其他进程。
- **内存分配方面**：系统在运行的时候会为每个进程分配不同的内存空间；而对线程而言，除了CPU外，系统不会为线程分配内存（线程所使用的资源来自其所属进程的资源），线程组之间只能共享资源
- **包含关系**：没有线程的进程可以看做是单线程的，如果一个进程内有多个线程，则执行过程不是一条线的，而是多条线
- **进程切换与线程切换**：
  + 进程切换开销更大，包括整个进程的地址空间、全局变量、文件描述符等
  + 线程切换只涉及到线程的堆栈、寄存器和程序计数器等，不涉及进程级别的资源



##### 进程的五种状态

 <img src="./../img/typora-user-images/image-20251106155950894.png" alt="image-20251106155950894" style="zoom: 50%;" />

- **NULL -> 创建状态**：一个新进程被创建时的第一个状态；
- **创建状态 -> 就绪状态**：当进程被创建完成并初始化后，一切就绪准备运行时，变为就绪状态，这个过程很快
- **就绪态 -> 运行状态**：处于就绪状态的进程被操作系统的进程调度器选中后，就分配给 CPU 正式运行该进程；
- **运行状态 -> 结束状态**：当进程已经运行完成或出错时，会被操作系统作结束状态处理；
- **运行状态 -> 就绪状态**：处于运行状态的进程在运行过程中，由于分配给它的运行时间片用完，操作系统会把该进程变为就绪态，接着从就绪态选中另外一个进程运行；
- **运行状态 -> 阻塞状态**：当进程请求某个事件且必须等待时，例如请求 I/O 事件；
- **阻塞状态 -> 就绪状态**：当进程要等待的事件完成时，它从阻塞状态变到就绪状态；



#### 用户态 vs 内核态

+ 内核态Kernel Mode: 权限更高，CPU可以执行所有的指令和访问所有的硬件资源。主要用于操作系统内核的运行
  内核态操作包括：**内存管理、进程管理、设备驱动程序控制、系统调用**等
+ 用户态User Mode: CPU只能执行部分指令集，无法直接访问硬件资源。主要用于运行用户程序

内核态和用户态的划分有助于保证操作系统的安全性、稳定性和易维护性

1. 防止恶意程序破坏系统资源
2. 用户态程序出现问题时危害整个系统



### Docker

##### Docker和虚拟机有什么区别？
 A: 虚拟机是虚拟出整个操作系统，比较重，启动慢，占资源多。Docker容器是共享宿主机的内核，只虚拟应用层，所以轻量很多，启动快，资源占用少。打个比方，虚拟机像是买了整套房子，Docker像是合租一个房间，都能住人但成本差很多。

##### Docker的基本命令
 A: 最常用的就几个：docker pull拉镜像，docker run启动容器，docker ps看正在运行的容器，docker stop停止容器，docker rm删除容器。还有docker build构建镜像，docker images查看本地镜像这些。

**Q: 什么是Dockerfile？**
 A: Dockerfile就是构建Docker镜像的配置文件，里面写了怎么搭建环境。比如FROM指定基础镜像，RUN执行命令安装软件，COPY复制文件，EXPOSE暴露端口等等。就像是装修房子的施工图纸。
