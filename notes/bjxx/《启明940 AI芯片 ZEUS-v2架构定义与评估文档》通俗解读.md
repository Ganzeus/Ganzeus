# 《启明940 AI芯片 ZEUS-v2架构定义与评估文档》通俗解读

## 封面与基本信息

这份文档来自一家叫"交叉信息核心技术研究院"下设的"前沿架构与智能芯片研究中心"。文档的主角是一款叫做**启明940（Qiming 940）**的AI芯片，文档版本是V0.2，写于2021年6月28日，负责人是谭展宏。

简单来说，这是一份内部的芯片设计规划书，详细描述了这颗芯片要怎么造、造成什么样、能做什么事。

## 第一章：产品背景

### 1.1 历史项目回顾与"岁星系列"产品定义

首先解释一下名字的由来。

**"启明"**是金星（Venus）的别名——在黎明时分出现在东方天空的那颗特别亮的星，象征着这个团队最初的设计项目。团队已经做过两款芯片了：**启明910**和**启明920**。

从**启明930**开始，芯片的定位升级了——它不再是实验性质的，而是真正用于商业落地的**产品级AI推理芯片**（所谓"推理"，是指AI模型训练完之后，拿去实际使用时的计算过程，区别于"训练"阶段）。

一颗完整的SoC芯片（**SoC，System on Chip，即"片上系统"，意思是把很多功能模块都集成在一颗芯片上**）包含了完成任务所需的基本模块，比如：
- **Host CPU**（主处理器，负责运行操作系统、调度任务的"大脑"）
- **DDR Controller & PHY**（内存控制器和物理接口，负责和外部内存芯片通信）
- **PCIe Controller & PHY**（高速外部接口，用于连接显卡、加速卡等外部设备）
- **视频解码**（把压缩的视频变回图像的模块）
- **常用低速接口**（I2C、SPI、UART等，用于连接各种小型外围设备）

关于芯片架构名字的变化：过去**NPU核**（**NPU，Neural Processing Unit，神经网络处理器，专门用来跑AI模型的计算核心**）一直叫做**MUSE**，这个名字原本的意思是"多粒度稀疏引擎"（**稀疏计算，是指利用AI模型中大量数值为零的特点来跳过无效运算，从而提升效率**）。但在MUSE第三版的设计过程中，团队决定放弃对稀疏性的支持，所以再叫"MUSE"就不太合适了，于是决定在下一次迭代时，把架构名字正式改为**ZEUS（宙斯）**。

ZEUS的全称是：**Zippy Engine for Universal AI Speedup**，意思是"快速的、通用的AI加速引擎"。它的设计目标是：通用化（能跑各种AI任务）、迭代友好（方便以后升级）、产品系列兼容性强（不同代产品能共用一套软件工具）、扩展性好（可以拼接多颗芯片扩大算力）。

### 1.2 SoC与Chiplet形态的协同研发

这一节讲的是团队在启明930项目中同时探索了三种不同的"芯片形态"，以及从中学到了什么。

先解释一个关键概念：**Chiplet（芯粒/小芯片）**。传统芯片是把所有功能做在一颗大硅片上（叫SoC）。而Chiplet的思路是把芯片拆成若干个小模块（每个小模块就是一个Chiplet），然后用特殊的封装技术把它们拼在一起，像拼积木一样。这样做的好处是灵活、便宜、提高良率（**良率，指一批芯片中合格品的比例，芯片越大、良率越低**）。

在启明930系列里，团队同时推进了三个方案：
1. **传统SoC方案**（就是一颗完整的大芯片，单个Artem核心，Artem是NPU核的代称）
2. **母版形式的Chiplet方案**（1个Artem + 4个MUSE核，通过一块"母版"把它们连在一起）
3. **Chiplet大规模扩展方案**（16个Artem互联）

通过这些探索，团队得出了三条经验：

**(a)** 基于母版的Chiplet形态，适合用来快速、低成本地做**算力密集型的特定任务加速板卡**（比如项目代号"Homo"的那种产品）。对于那些目前还在用FPGA（**FPGA，Field Programmable Gate Array，现场可编程门阵列，一种可以灵活编程的通用硬件，性能不如专用芯片但开发周期短**）做加速卡的公司，如果他们想做成ASIC（**ASIC，Application Specific Integrated Circuit，专用集成电路，就是为特定用途定制的芯片，性能更高、成本更低但不灵活**），Chiplet是个不错的选择。**但有个限制：这种方案目前只能外挂一种Chiplet，也就是只能拼接同一种核心（同构）。**

**(b)** 大规模Chiplet扩展（比如16个核互联）被证明短期内不适合量产，主要原因是供应链成本高、良率差。结论是：每个Chiplet的颗粒度应该做大一些，Chiplet的数量应该尽量少。

**(c)** 传统SoC形态适合作为主线产品，稳健地一代代迭代，同时把NPU核做精、把配套的软件工具链（**工具链，是指编译器、调试器、运行库等一整套辅助开发工具**）做成熟。

**因此，启明940的策略是：**
- 主线继续做SoC，增大算力、增强通用性；
- 同时预留一路**D2D接口**（**D2D，Die-to-Die，即裸片到裸片的高速直连接口**）继续探索Chiplet：
  - 方案一：外挂一颗CPU裸片，尝试"NPU+CPU分开封装"的集成方案（原因是AI产品里NPU和CPU的更新频率不同，NPU一般每两次更新一次，CPU才更新一次，分开可以让两者独立迭代）；
  - 方案二：两颗NPU互联，将两个4×4的核心网格拼成一个4×8的更大系统，让它们在使用上看起来像一个完整系统，而不是像之前16-Artem那样基本各自独立工作。

对于基于Artem-v1母版的Chiplet方案，会继续服务于**Chiplet Design House**（Chiplet设计公司，专门设计和销售Chiplet模块的公司）的产品线，后续会对Artem-v1进行优化。

## 第二章：场景定义

启明940是一颗**通用高算力AI芯片**，它有一定的接口扩展性，可以配合第三方芯片做成不同形态的产品，包括但不限于：AI推理加速板卡、AI工控机（工业控制计算机）盒子、机器人、无人车等。主要目标是覆盖以**视觉任务为主、语音文本任务为辅**的AI应用场景。

### 三种使用模式

**模式一：独立SoC模式（主处理器模式）**

芯片自己就是整个系统的"大脑"，里面的CPU负责运行操作系统和调度AI任务。就像一台小型计算机，启明940自己就能独立完成一整套AI工作流程。

**模式二：协加速器模式（协处理器模式）**

系统里有一颗更强的外部主CPU（比如服务器的CPU），启明940只负责完成AI计算加速的部分，相当于一个"AI计算帮手"。

**模式三：PCIe从设备模式**

启明940作为一块类似显卡的PCIe插卡（**PCIe，Peripheral Component Interconnect Express，一种高速插槽标准，显卡、固态硬盘等都用这个接口连接主板**），插到服务器或PC上，主机通过PCIe接口给它发数据，它处理完再返回结果。

### 图像输入的两种方案

**方案一（带ISP）：**
摄像头的原始数据通过**MIPI接口**（**MIPI，一种专为移动设备设计的高速串行接口，常用于连接摄像头**）和串行解串器（**SerDes，Serializer/Deserializer，一种把并行数据转成串行传输、再还原的器件，用于长距离传输**）传到SoC，SoC里的**ISP**（**ISP，Image Signal Processor，图像信号处理器，负责把摄像头的原始数据转换成漂亮的彩色图像**）做完图像处理后直接进行AI推理。这种方案更适合端侧应用，比如直接做进摄像头模组里。

**方案二（不带ISP）：**
摄像头模组里加一颗专用的图像处理芯片，在那边做完ISP处理后进行视频编码，然后通过**千兆网口**把数据传到主SoC芯片上解码再做AI推理。这种方案更通用，适合边缘计算场景（**边缘计算，是指在靠近数据产生地的地方进行计算，而不是把所有数据发到远程云端**）。文档里提到华为的**昇腾310**也是这种方案。

最后还提到了**云边协同**（云端和边缘端协作完成AI任务）的应用场景。

## 第三章：SoC与ZEUS架构规格定义

### 3.1 SoC整体规格说明

这一节是一张表格，列出了启明940芯片的核心指标，我来逐项解释：

**算力**

- **Uni-INT4 (8A8W)：64 TOPS**。所谓INT4是指用4位整数（**量化精度，用更少的比特位来表示数字，牺牲一点精度换取更高速度和更低功耗**）来表示权重，8A4W的意思是激活值用8位、权重用4位。64 **TOPS**（**TOPS，Tera Operations Per Second，每秒万亿次操作，是衡量AI芯片算力的单位**）是峰值算力。
- **INT8 (8A8W)：32 TOPS**。激活和权重都用8位整数，32TOPS。
- **Uni-INT16 (16A8W)：16 TOPS**。激活16位、权重8位。
- **FP16 (16A16W)：16 TOPS**。用半精度浮点数（**FP16，16位浮点数，精度比INT8高但算力低**），16TOPS。

**内存带宽：1066 Gb/s**，使用256位宽的LPDDR4X物理接口（**LPDDR4X，一种低功耗双倍速率内存，常用于手机和嵌入式设备**）。

**内存规格：LPDDR4X，容量8GB到32GB可选。**

**芯片面积：**裸片（**Die，从晶圆上切下来的未封装芯片本体**）约300mm²，封装后约1089mm²（大约33mm×33mm，跟一枚50块钱硬币差不多大）。

**工艺：TSMC 12FFC**（台积电12纳米FinFET Compact工艺，**纳米制程越小，晶体管越小，同样面积能放下更多晶体管，性能更强、功耗更低**）。

**典型频率：CPU 1.8-2.1 GHz，NPU 1 GHz。**

**典型功耗：预计30-50W**（待定，芯片本身完整功耗）。

**NPU能效：目标2 TOPS/W，至少1 TOPS/W**（意思是每消耗1瓦电能完成多少万亿次AI运算）。

**Host CPU（主处理器）能力：** 使用16核ARM A76（**ARM A76，ARM公司设计的一款高性能处理器核心，常见于手机芯片**），主频1.8-2.1 GHz，负责运行简单操作系统、调度AI前后处理任务。

**MCU（微控制器）能力：** 使用ARM China星辰系列MCU（**MCU，Microcontroller Unit，微控制器，是一种低功耗小型处理器，负责系统唤醒、低功耗管理等轻量级控制任务**），负责唤醒、低功耗处理和宏观控制。

**视频处理能力：** 使用ARM LINLON-V5 IP（**IP，Intellectual Property，知识产权模块，指向第三方购买的现成功能模块**），支持4K@240fps H.264/H.265解码（**4K@240fps，指4K分辨率每秒240帧，非常高的视频规格**），以及4K@60fps编码。

**存储接口：** SD6.0/eMMC5.1（**eMMC，嵌入式闪存，常见于手机、平板**）和QSPI Flash（**QSPI Flash，一种小容量快速闪存，通常用于存放芯片的启动程序**）。

**高速接口：**
- **PCIe 4.0 x16**（**PCIe，高速外部设备接口，x16表示16条数据通道，PCIe 4.0是当时较新的版本，带宽约256 Gbps**）
- **USB 3.1-C**（高速USB接口）
- **RGMII x2**（**RGMII，一种以太网物理接口标准，连接千兆网络芯片**）

**低速接口：** I2C、SPI、UART、I2S、GPIO等（**这些都是常见的嵌入式通信接口，用于连接传感器、显示屏、音频设备等小型外设**）。

**片上SRAM：34 MB**（**SRAM，Static Random Access Memory，静态随机存储器，速度极快但面积较大，作为NPU计算时的片上缓存**）。

**主要面向任务：** CNN（卷积神经网络）、MLP（多层感知机）、RNN（循环神经网络）、LSTM（长短期记忆网络），能支持Apollo公开库、MLPerf基准测试（**MLPerf，行业权威的AI性能基准测试**）和AIIA基准内的模型。

**预计流片时间：2023年2月底**（**流片，即正式送到晶圆厂生产芯片，Full Mask意为完整的光刻掩膜版，是正式量产的那种，区别于试验性的MPW**）。

### 功能指标要求详解

**SoC层面要求：**

1. 支持独立SoC模式（片内CPU当主CPU）和协加速器模式（外挂CPU，片内CPU可以关掉）。
2. 独立SoC模式下，CPU要能跑基础Linux系统，完成图像分类、检测、分割、语音识别等AI任务的完整流程（包括前处理和后处理）。
3. 需要有视频和JPEG的编解码能力（**编解码，Codec，指视频的压缩和解压缩，节省存储空间和传输带宽**）。
4. ISP不集成在片内，用片外芯片解决即可。
5. 支持低功耗休眠和唤醒（但这版暂不考虑**DVFS，动态电压频率调节，一种根据负载自动调整电压和频率来省电的技术**）。
6. 预留一路D2D接口，用于两种Chiplet实验：①两颗SoC裸片拼成一个4×8的更大NPU网格；②外挂CPU裸片实现NPU和CPU分芯片互联。对于内部互联方式，考虑了两种思路：一是把所有模块都用2D-Mesh网格直接互联（**2D-Mesh，二维网格网络，一种芯片内部各模块之间互相连接的拓扑结构，像棋盘格一样**）；二是NPU挂在一条总线上，NPU内部用2D-Mesh。
7. 支持各种常见高低速接口。

**NPU层面要求：**

1. Weight Buffer（**权重缓存，用于暂存神经网络的模型参数**）要尽可能大，能放下大多数网络峰值存储量的那一层数据（避免频繁从片外内存搬运数据）。
2. 对于小网络，能实现**Layer-Pipeline**（**层流水线，指前一层的输出数据还没全部计算完，就开始把已经完成的部分送给下一层处理，提高利用率**）。
3. 考虑支持2~4个不同模型同时并行计算（**多模型并发推理**）。

### 3.2 SoC架构（互联方案选择）

这一节解决的是芯片内部各个模块之间"怎么连"的问题。

团队考虑了两种互联方案：

**方案一：全Mesh NoC互联**
把所有模块都用2D网格网络（**NoC，Network on Chip，片上网络，是芯片内部各模块之间传输数据的"公路网"**）连起来。好处是扩展性好，尤其方便Chiplet扩展——加了新的芯粒，整个系统的变化对软件来说几乎是透明的（感觉不到）。缺点是实现复杂、难度大。

**方案二：Bus + Mesh层次化互联**
CPU和外围设备挂在一条**AXI总线**（**AXI，Advanced eXtensible Interface，ARM公司定义的高性能片上总线协议**）上，NPU内部用Mesh网格。好处是简单、通用，CPU和NPU分离也更方便两种Chiplet方案的实现。

**最终决定采用方案二。** AXI Bus选ARM或Arteries的总线IP，AI NoC暂定Arteries的AI Package NoC IP，基于AXI4协议。

整个SoC系统被划分为**10个子系统**：MCU子系统、CPU子系统、AI加速子系统、D2D高速互联子系统、高速接口子系统、编解码+ISP子系统、存储子系统、其他低速外设接口、SOC系统总线。

---

#### 3.2.1 MCU子系统

MCU负责两件事：**系统控制**和**低功耗管理**。
具体就是对各子系统的配置寄存器（**寄存器，芯片内部用于存放控制参数的小型存储单元**）进行管理、配置各个模块的低功耗状态，以及控制外围接口。使用ARM-China星辰系列MCU。

---

#### 3.2.2 CPU子系统

CPU负责两件事：**运行操作系统**和**执行算法任务**。

运行操作系统主要用于独立SoC模式，让这颗芯片能像一台小电脑一样独立工作。

执行算法任务主要是处理那些NPU不擅长做的工作，比如神经网络的**前处理**（把原始数据变成NPU能处理的格式）和**后处理**（把NPU的输出结果变成有意义的信息，比如画出检测框）。

由于实际需要CPU做多少计算还不好评估，团队暂定用**20核A76**来搭建CPU子系统（比规格表里说的16核还多了4核，可能后续调整了）。

---

#### 3.2.3 编解码+ISP子系统

编解码（Codec）主要做视频解码。具体需求：
- 支持H.264、H.265解码，支持JPEG、PNG图像格式；
- 解码性能：1080P @960fps（相当于支持16路1080P @60fps同时解码）；
- 编码性能：1080P @240fps（相当于支持4路1080P @60fps同时编码）；

ISP部分支持8M像素@60fps的图像处理（**8M，即800万像素，相当于4K分辨率的摄像头**）。

---

#### 3.2.4 AI加速子系统

这是芯片最核心的部分，负责完成绝大多数AI计算（conv2d卷积、maxpool/avgpool池化、upsample上采样、BN批归一化等）。

需求：
1. 支持32TOPS算力；
2. 能通过片间扩展组成64TOPS算力（两颗芯片联用）；

**具体架构：**
- 由**16个ZEUS Core**组成；
- 加上**四组D2D接口**用于片间扩展；
- 这些模块通过一个**4×6 Mesh NoC**连接在一起。

为什么是4×6而不是4×4？因为需要把16个ZEUS Core用4×4的网格连接，而Arteries的AI NoC每个路由节点（**Router Node，网格上的交叉点，相当于路口，负责把数据转发到正确方向**）只能连一个主设备和一个从设备，D2D接口和SoC通信端口也需要连进来，所以要扩展成6×6 Mesh NoC（多出的节点可以被EDA工具自动优化掉）。

同时，这个NoC需要支持**广播和组播**（**广播：把同一份数据同时发给所有节点；组播：同时发给一组特定节点；区别于单播，单播是一对一发送**），最好能实现all-to-all（任意节点到任意节点）的广播和组播。

---

#### 3.2.4 存储子系统

存储子系统为整个SoC提供数据存储，包含以下几种存储：

**LPDDR4X**：系统主内存，CPU跑操作系统和软件时用的。总带宽为4266 Mbps × 32bit × 8组 = **1066.5 Gbps**。

文档还给出了一张详细的带宽需求计算表（以1080P@960fps为例）：
- 编码视频流输入带宽：约3.75 Gbps
- 解码后得到原始图像：约44.5 Gbps
- CPU进行AI前处理：约89 Gbps（读+写各44.5 Gbps）
- NPU进行AI计算（估算为原始图像带宽×5）：约445 Gbps（读+写各222.5 Gbps）
- 总计读+写：约**586 Gbps**

这意味着LPDDR4X的1066 Gbps的带宽虽然看起来很大，但在高负载时实际上还是很紧张的。

**SRAM**：片上小容量高速存储，用于系统调试和启动。
**QSPI Flash**：存放启动程序（**Bootloader，引导加载程序，类似于电脑的BIOS**）。
**SD/eMMC**：外接大容量存储，存放操作系统文件等。

### Chiplet方案1：CPU+NPU分裸片

这个方案是把CPU和NPU放在两颗不同的裸片上，通过**Innolink**（一种高速片间互联接口，**类似于英特尔的EMIB或AMD的IF总线，专门用于芯粒之间高速通信**）连接：
- CPU+Bus那颗裸片：屏蔽掉片内NPU；
- NPU那颗裸片：屏蔽掉非NPU部分；

数据流向举例：
- **NPU配置路径**：CPU → AXI总线 → Router → Innolink → Innolink → NPU Mesh → NPU Core
- **数据输入路径**：DMA/Codec/PCIe → AXI总线 → Router → Innolink → Innolink → NPU Mesh → 片上存储
- **数据输出路径**：从NPU存储 → 逆向路径 → 返回CPU侧的LPDDR4/PCIe

---

### Chiplet方案2：4×8 NPU Mesh

这个方案是把两颗SoC裸片的NPU部分拼在一起，构成一个4×8（共32核）的更大NPU系统。两颗裸片上的非NPU部分只有一颗正常工作，另一颗全部屏蔽，两个NPU Mesh通过Innolink互联。

这个方案的挑战在于：如何充分、均衡地利用两颗裸片上的存储（GDDR6/SRAM）带宽，避免有些存储单元很忙而其他很闲，从而造成性能下降。

### 各典型使用场景的数据流描述（3.5节）

这部分描述了启明940芯片在各种实际使用情景下，数据是怎么流动的。

- **MCU启动**：MCU从QSPI Flash/EMMC/SRAM/LPDDR4中读取启动代码，完成系统初始化。
- **PCIe数据交互**：CPU控制PCIe控制器，数据在PCIe外部设备和LPDDR4之间双向流动。
- **USB数据交互**：类似PCIe，CPU控制USB，数据存入/读出LPDDR4。
- **以太网数据交互**：外部网络 → 以太网MAC → LPDDR4，或反向。
- **摄像头输入**：MIPI接口接收摄像头数据 → 存入LPDDR4。
- **视频编解码**：LPDDR4 → CODEC → LPDDR4（解压缩视频帧或压缩视频帧，存回内存）。
- **AI计算**：LPDDR4中的数据 → 经过CPU前处理 → 送入NPU计算 → 结果回LPDDR4 → CPU后处理。
- **DMA数据搬运**（**DMA，Direct Memory Access，直接内存访问，是一种不经过CPU就能在内存和外设之间搬运数据的机制，解放CPU算力**）：在各子系统和各类存储之间自动搬运数据，不占用CPU。
- **Chiplet间数据搬运和AI计算**：数据通过Innolink在两颗裸片的NPU之间双向传输，协同完成AI计算。

## 第四章：ZEUS-v2架构具体定义

### 4.1/4.2 量化方案

这一章是最技术性的部分之一，讲的是芯片如何处理AI模型中的数字精度问题。

#### 4.2.1 非对称量化

为了让芯片跑得更快、更省电，我们不能把神经网络里所有数字都用32位浮点数（FP32）表示——那太占资源了。我们需要把它们压缩成8位整数（INT8）甚至4位整数（INT4），这个过程就叫**量化**。

相比**对称量化**（把数字范围对称地映射到±127），**非对称量化**通过引入一个**缩放因子（scale）**和一个**零点（zero point）**，能更灵活地把原始FP32数字的最小值/最大值分别映射到INT8的最小值/最大值。这样对于数据分布不均匀的情况处理得更好。

举个例子：想象你要把一根-0.1到3.5之间的橡皮筋，均匀压缩到一把0到255刻度的尺子上。非对称量化就是把橡皮筋的-0.1对应到尺子的0，3.5对应到255，中间线性插值；而对称量化则是把0对应到尺子的中间（128），两边对称，这样-0.1到3.5的范围可能就没有被充分利用。

量化公式：量化后的数 = 原始数 / scale + zero_point

对NPU来说，计算时需要额外建立处理zero_point的累加通路。

#### 4.2.2 量化粒度

**张量级量化（Tensor-wise）**：整个tensor（特征图）只用一个scale，粗粒度，简单但精度损失相对大。

**通道级量化（Channel-wise）**：每个输出通道用一个独立的scale，细粒度，精度更高但参数更多。

对于**Weight（权重）**：采用通道级量化（因为不同通道的权重分布差异较大，用一个scale会损失精度）。

对于**Activation（激活值，也就是每一层计算的中间结果）**：采用张量级量化（因为经过BN层处理后各通道分布已经比较均匀，通道级量化没有明显收益，而且参数更少，更简单）。

#### 4.2.3 量化取整方式

把浮点数量化成整数时，小数部分怎么处理？有三种方式：
- **Ceil**：向上取整（1.3 → 2）
- **Floor**：向下取整（1.7 → 1）
- **Round**：四舍五入（1.5 → 2）

从表格数据可以看出，对于PTQ（**PTQ，Post-Training Quantization，训练后量化，即模型训练完之后再做量化，不需要重新训练**），Round方式最好（ResNet50在ImageNet上的精度：Round=76.01%，Ceil=74.46%，Floor=74.28%）。

因此ZEUS-v2需要支持三种取整方式。文档还特别指出，之前的**Jupiter-910不支持Round**，这是需要在新芯片上修复的问题。

#### 4.2.4 Element-wise量化

**Element-wise操作**（**逐元素操作，指把两个相同形状的tensor对应位置的元素进行运算，比如ResNet里的残差相加就是Element-wise Add**）在量化时遇到一个难题：两路输入的scale可能不一样。

**Element-wise Add（加法）的处理：**

设输入A的scale为scale\_A，输入B的scale为scale\_B，输出的scale为scale\_O，各有自己的zero\_point（zp）。

由于两路scale不同，不能直接把两个INT8整数相加，需要先把其中一路乘以一个浮点系数（scale\_B/scale\_A）来对齐到同一个scale，然后再相加。核心变成一个浮点乘法加一个浮点加法。

**Element-wise Mul（乘法）的处理：**

同理，由于两路scale和zero\_point都不同，不能直接做INT8乘法。需要展开计算，变成整数加法和整数乘法的组合（两个int加法+一个int乘法，或三个int乘法+两个int加法）。

### ZEUS架构说明（本章末尾）

除了上述要求，启明940的ZEUS架构还需要实现以下设计目标：

1. 预留Innolink接口，能把2个ZEUS-v2联成4×8的多核系统，这个系统在使用上要像一个完整独立的系统（而不是两个独立的系统）；
2. Innolink接口也能接外挂CPU；
3. 架构具备可扩展性——从这代开始，后续迭代主要是扩大Core规模：940是4×4，950规划为8×8；
4. 指令具有**向后兼容性**（**旧软件能在新硬件上运行**），编程方便，软件和硬件解耦清晰；
5. 一共16个核心，每个核心中加入一个**通用向量处理器（Vector Processor）**，统一处理Pooling（池化）、Padding（补零）、Upsampling（上采样）、Element-wise（逐元素运算）、Shuffle（通道重排）等操作。

**ZEUS-v2的宏观架构**分两个层次：

**Tensor Core级别（顶层）：**
顶层是由4×4个**Tensor Core**（相当于之前的MUSE核，每个Tensor Core是一个完整的计算单元）构成的2D-Mesh。全局共享的存储是**L3 Unified Shared Buffer**（**L3，三级缓存，在多个Tensor Core之间共享的大容量缓存，比L2更大但速度稍慢**），构成类似**NUMA**（**NUMA，Non-Uniform Memory Access，非均匀内存访问，是指离某个处理单元近的内存访问速度快、远的慢**）的存储结构。

**PE级别（单个Tensor Core内部）：**
每个Tensor Core内部是4×4的PE（**PE，Processing Element，处理单元，负责具体做乘加运算的最小计算单元**）阵列（旧版是8个PE全相连）。每个Tensor Core还新增了一个**Vector Processor**，统一处理各种非卷积计算。

**相比启明930，Tensor Core的主要改动：**
1. PE中每个lane（**lane，一个并行计算通道**）从16个减少到8个；
2. 量化方式从对称量化改为非对称量化；
3. 每个lane新增**BF16**（**BF16，Brain Float 16，一种16位浮点格式，由Google提出，保留了FP32的指数范围但降低了精度，适合深度学习**）乘累加通路；
4. PE中加入组卷积功能（用mask实现）；
5. PE之间不再共享activation/weight的L1 Buffer（**L1，一级缓存，PE内部最近的缓存，速度最快**）；
6. Weight Memory大小改为9KB/lane；
7. Activation L1 Buffer扩容，最大可存储2×2×1024×8bit，支持**Ping/Pong**（**Ping/Pong，双缓冲机制，一块缓冲区在计算的同时另一块在读/写新数据，避免相互等待**）；
8. Out Buffer改为16×8 pixel，支持Ping/Pong；
9. 反卷积中对零值计算的优化；
10. Kernel size和stride支持水平/垂直方向不同的值；
11. PE之间存在切input channel（**把输入通道切分给多个PE并行计算**），有4个PE需要传递和累加**部分和（Partial Sum/Psum）**（**Psum，即中间累加结果，当输入通道数太多、一个PE算不完时，多个PE各算一部分然后把结果加在一起**）；
12. PE阵列支持6种数据映射模式（WH、WC、NH、NC、KH、KC，后面详细解释）；
13. L2考虑使用line buffer。

### 任务块映射说明

这部分讲的是如何把神经网络卷积计算分配到PE阵列上并行执行，这是AI芯片设计中最核心的问题之一。

首先定义几个字母：R=卷积核高度、S=卷积核宽度、H=输出特征图高度、W=输出特征图宽度、C=输入通道数、K=输出通道数、N=Batch大小、X=PE阵列横向、Y=PE阵列纵向。

**PE内MAC阵列的空间映射：**
- X方向固定映射C（输入通道）；
- Y方向固定映射K（输出通道）；

不同精度模式下的并行度：
- **8A4W（激活8bit、权重4bit）**：Y方向8个输入通道，X方向16个输出通道，峰值算力最高（64 TOPS）；
- **8A8W（INT8×INT8）**：Y方向8个，X方向8个（32 TOPS）；
- **16A8W、16A16W（BF16）**：Y方向4个，X方向8个（16 TOPS）；

**时间维度的折叠：**
由于一个卷积层的计算量往往远超PE阵列的容量，必须把整个计算任务切成很多小块，按顺序喂给PE阵列。这就涉及多级缓存的协同：
- **AL1/AL2**（**激活值的一级和二级缓存，AL1在PE内部，AL2在PE阵列级别**）：由于输入特征图各计算块之间有重叠，所以要分两级缓存；
- **OL1/OL2**（**输出值的一级和二级缓存**）：因为部分和的位宽比最终输出更大，也需要两级；
- **WL1**（**权重一级缓存**）：权重使用规律较简单，只需一级；

**数据更新策略（红色部分，L1缓存更新顺序）：**
支持4种模式：K-N-H-W、K-N-W-H、N-H-W-K、N-W-H-K（这里的字母顺序代表在L1缓存中更新数据时，沿哪个维度先迭代）。

**数据复用模式（蓝色部分）：**
采用**WS（Weight Stationary，权重固定）**模式（把权重留在缓存里不动，不断换输入数据过来乘）。

**空间并行层次（紫色部分）：**
- MAC阵列：固定映射为XY-KC（X对应K输出通道，Y对应C输入通道）；
- PE阵列：提供6种不同映射模式；
- Core阵列：提供5种模式，预留多任务/多模型部署空间。

**存储约束公式**解释：文档给出了一系列公式，确保每次切块的数据量不超过各级缓存的容量，这是编译器在生成执行计划时必须满足的约束条件。

---

### 9种Partition（分块）模式的分类

文档描述了将任务分配给4×4核心阵列的9种方式，把它们归纳为三类：

**I类（权重相同，广播权重）：** 所有Core的weight相同，需要广播/组播权重数据；每个Core的activation不同（各自持有一部分，但通道是完整的）。适合权重数据量小的层。

**II类（激活值相同，广播激活）：** 所有Core的weight都不同，激活值相同，需要广播激活值。适合权重数据量大的层。

**III类（部分相同，组播）：** 部分Core的weight相同，部分Core的activation相同，都需要组播。适合中等weight数据量的层。

### 4.5 数据格式

这一节讲芯片里的数据是怎么存放的，涉及三种数据：activation（激活值）、weight（权重）、以及BN/激活等层的参数。

#### 4.5.1 Activation数据

Activation数据就是每一层网络的输入特征图。

**坐标表示：** 用[N, H, W, C]表示（N=Batch数量、H=高度、W=宽度、C=通道数），用[n, h, w, c]表示具体某个点的位置。

**数据格式：** 三种——uint8（8位无符号整数，量化后）、uint16（16位无符号整数，量化后）、FP16（16位浮点，不需要量化）。

**在DDR（外部内存）中的排布：** 按C→W→H→N的顺序密排（对齐到算法框架如PyTorch的存储方式）。举例：4×4空间、68个通道的数据，地址0~67存放[h=0,w=0]处所有通道，地址68~135存放[h=0,w=1]处所有通道，以此类推。

**在L2 Buffer（片上缓存）中的排布：** 同样按C→W→H→N顺序，但需要**对齐**——uint8需要channel数对齐到8的整数倍，uint16/FP16需要对齐到4的整数倍。比如68个channel要补零到72（68→72）。这是为了方便PE阵列做64bit并行读取。

#### 4.5.2 Weight数据

Weight数据是卷积核/全连接层的参数，在模型训练完后就固定了，推理时不变，所以可以提前由编译器一次性整理好格式，不需要运行时动态重排。

**坐标表示：** 用[K, C, S, R]（K=输出通道数、C=输入通道数、S=卷积核宽度、R=卷积核高度）。

**数据格式：** 三种——uint4（4位整数量化）、uint8（8位整数量化）、FP16。

Weight数据在DDR和L2 Buffer中存储方式是一样的，都按计算友好型排布：按C0→R→S→C1→K的顺序，C0是为了对齐到8 byte（C0×C1等于对齐后的C）。

**int8 weight的存储：** 以8个channel为一组，先排完一个RS平面的所有点的前8个通道，再排第8~15个通道……不足8个通道的补零。

**int4 weight的存储：** 因为只支持8bit激活×4bit权重模式，PE内部做64bit×64bit计算时需要8个激活通道和16个权重通道，所以把两个输出通道的int4数据拼成1个byte，再按int8方式排布。

**FP16 weight的存储：** 每4个channel为一组（因为FP16是2 bytes，只能放4个通道而不是8个），其余与int8类似。

**Weight在WL1（片内权重缓存）中的存储：** WL1分布在每个PE中，分为8组，对应8个输出通道组。总的1184KB分布在16个PE中，具体分布方式取决于6种PE阵列映射模式中的哪一种。

#### 4.5.3 其他参数数据

除了权重，还有一些层需要额外参数：

**Group Conv的Mask参数：**
组卷积中，不同输出通道只需要部分输入通道参与计算。为了告诉硬件哪些输入通道需要参与，用一个**Mask**（掩码，每个bit代表一个输入通道是否参与计算）来表示。这个Mask和weight数据存放在一起，需要做256bit对齐（不足256bit的补零）。

**反量化参数（Dequantization）：**
计算完卷积后，结果是整数，需要还原成浮点数的近似值，这个过程叫反量化。反量化公式是：X = K\_dq × Xq + B\_dq。

对于单独的activation反量化：K\_dq和B\_dq只有一组（张量级），可以通过指令给出。

对于卷积的反量化（需要考虑weight的量化误差）：展开计算后，K\_dq = scale\_A × scale\_W，B\_dq = scale\_W × zp\_A × Σ(Wq)。由于W是通道级量化，每个输出通道的K\_dq和B\_dq都不同，需要单独存储（如图所示，按输出通道排列的参数数组）。

**BN（批归一化）参数：**
BN可以等效为Y = K\_BN × X + B\_BN，其中K\_BN和B\_BN是每个输出通道一个值（不同通道值不同），需要存储整个通道数量的参数数组。

**激活函数参数：** 分四类——①不需要参数（如ReLU）；②参数固化在硬件逻辑中（如Tanh、Sigmoid）；③只需一个参数可通过指令给出（如ReLU-X、LeakyReLU）；④需要按通道传参（如PReLU，需要实时读取）。

**量化参数：** 反量化的逆过程，Y\_q = K\_q × Y + B\_q，K\_q和B\_q是张量级的，只需一组，通过指令给出。

**反量化+BN融合：**
观察到反量化（Y = K\_dq × Xq + B\_dq）和BN（Z = K\_BN × Y + B\_BN）形式相同，可以串联合并为一步：Z = (K\_BN × K\_dq) × Xq + (K\_BN × B\_dq + B\_BN)，把"乘+加+乘+加"变成"乘+加"，节省硬件资源和计算时间。

**卷积+反量化+BN+激活+量化的全融合：**
ZEUS-v2把这整条处理链都融合成一个操作，激活/量化参数通过指令给出，BN参数和weight数据合并存放，组成最终的model parameter（模型参数包）。对于普通卷积、空洞卷积、反卷积，没有mask；对于组卷积，mask也合并进去。

最终两种形式：
- **非组卷积的模型参数排布：** weight数据 + BN参数（K\_BN、B\_BN各一组，每输出通道一个值）+ 反量化参数；
- **组卷积的模型参数排布：** 在上述基础上，再加上group conv的mask数据。

### 4.5（NPU指令架构方案）

#### 4.5.1 指令类型与流程

NPU的任务处理分六个阶段：**取指 → 译码 → 数据加载 → 计算 → 存储 → 同步**。

**数据加载指令：**
- **LW**（Load to WL1）：从DDR或本地L2缓存加载weight数据到WL1（PE内的权重缓存）；
- **LD**（Load to L2）：从DDR加载activation/weight数据到本地L2缓存；
- **SI**（Sync In）：从远端另一个NPU Core的L2缓存同步activation/weight数据到本地L2；

**计算指令：**
- **DT**（Data Transform）：从L2读数据，做维度转换，结果写回L2；
- **PE**（PE Tensor Calculate）：从L2读activation到AL1，从WL1读weight，做卷积/FC计算，把OL1中的结果写回L2；
- **VP**（Vector Processor）：从L2读activation，做池化等操作；

**存储指令：**
- **ST**（Store from L2）：把L2数据写回DDR；
- **SO**（Sync Out from L2）：把本地L2数据同步到远端另一个NPU Core的L2；

#### 4.5.2 多任务乱序发射

ZEUS-v2采用**多任务指令队列**，每类指令有独立的队列，可以并行处理。

**指令间依赖问题的解决方案——信箱机制：**

想象这样一个场景：PE指令正在计算，ST指令想要把结果存到DDR，但结果还没算完，ST必须等。但与此同时，LD指令可以先去加载下一批数据，两者并行。

为了自动管理这些依赖关系，每条指令都带有两类标记：
- **收信位（前向依赖标记）**：9个bit，表示"我发射之前，我依赖哪些类型的指令先完成"；
- **寄信位（后向通知标记）**：9个bit，表示"我执行完成后，通知哪些类型的指令可以继续"；

硬件里有**9×9个指令信箱计数器**（9种指令类型两两之间各一个信箱）。比如"PE→ST信箱"：PE指令完成时+1，ST指令发射时检查这个信箱，>0才能发射并-1，否则等待。这样就用硬件信箱实现了细粒度的指令间依赖追踪，最大化指令并行度。

---

#### 4.5.3 LD指令格式

LD指令用于把DDR中的一块数据搬到L2缓存中。

指令格式：`LD DTile[DTaddr, DTW, DTH, DTC, DTN] STile[STaddr, STW, STH, STC, STN] DPos[DPW, DPH, DPC, DPN]`

- **STile（源数据块）**：DDR中的大数据块，用起始地址、宽、高、通道数、batch数描述；
- **DTile（目标数据块）**：要读取到L2中的那一小块，用起始地址、宽、高、通道数、batch数描述；
- **DPos（目标块在源块中的位置）**：DTile在STile中的偏移位置（水平、垂直、通道、batch方向各一个坐标）；

这种设计允许灵活地从一块大数据中"切出"任意位置的一块，方便实现卷积的滑窗读取、padding处理等。

### NPU硬件架构（各模块详解）

#### LOAD DMA（加载数据传输控制器）

LOAD DMA负责从外部（DDR或其他NPU Core）把数据搬到片内。

**接口：**
- 一组**AXI4 Master Read接口**（**AXI4 Master，主动发起读请求，从外部抓数据**）：主动从DDR读activation/weight/指令数据；
- 一组**AXI4 Slave Write接口**（**AXI4 Slave，被动接收外部写进来的数据**）：接收其他NPU Core或通过Innolink发来的数据；
- 与L2 Buffer和PE-WL1的数据接口均为256bit宽；
- AXI接口运行在axi_clk时钟域，与内部模块接口运行在core_clk时钟域（**跨时钟域处理，需要同步器**）；

**功能亮点：**
- 支持1024bit非对齐访问（**非对齐访问，指数据的起始地址不是总线宽度的整数倍，处理起来更复杂但更灵活**）；
- 支持分段循环处理（指令执行时只能访问L2的一段空间，超过结束地址自动回绕到起始地址，类似**环形缓冲区**）；
- 支持SIA/SIP同步指令（**用于多Core间的数据同步：SIA是被动等待对方发来数据，SIP是主动通知对方来取数据**）；

#### STORE DMA（存储数据传输控制器）

STORE DMA负责把L2缓存的数据写回DDR或发给其他NPU Core。

**接口：**
- 一组**AXI4 Master Write接口**：主动把L2数据写到DDR或发给其他Core；
- 一组**AXI4 Slave Read接口**：接受其他Core发来的读请求，从本地L2取数据发给对方；

支持与LOAD DMA类似的分段循环访问、SOA/SOP同步机制（对应LOAD DMA的SIA/SIP）。

#### Inst Decode（指令译码器）

负责缓存从LOAD DMA收到的指令数据，做初步解析，把原始指令代码变成各执行单元能理解的任务命令。

#### Inst Queue（指令队列与调度器）

收到译码后的任务指令，根据类型分发到对应的执行单元队列，同时：
- 维护9×9个信箱计数器，管理指令间依赖；
- 根据依赖关系决定指令何时可以发射；
- 支持**Debug模式**（可以单步执行指令、设置断点、暂停/继续、从debug口注入指令）；
- 支持**Trace模式**（记录每条指令的内容、分发时刻、发射时刻、完成时刻、时间戳，方便性能分析和调试）；

#### L2 Buffer（二级缓存）

L2 Buffer是PE阵列和Vector Processor的本地缓存，容量**1MB**，存储activation和weight数据，对AXI Slave接口可见（意味着其他Core可以通过总线直接读写这里的数据）。

由多个双端口SRAM（**tpram，支持同时读和写两个操作**）组成，每个RAM宽64bit，总容量1MB。

提供超过**10路**并发读写端口：LW用1路256bit读、LD用1路256bit写、SI用1路256bit写、PE读控制器用4路64bit读、PE写控制器用4路64bit读写、VP用1路256bit读写、DT用1路256bit读写、ST/SO/Debug各1路256bit读。

设计时需保证大于5×256×2 Gbps的写带宽和7×256×2 Gbps的读带宽。同时计划支持**ECC**（**Error Correcting Code，错误纠正码，能检测并自动纠正单bit翻转错误，提高可靠性**）。

#### DT（Data Transform，维度转换模块）

负责对L2中的四维矩阵做维度变换，比如把(N,C,H,W)转置成(N,C,W,H)。支持INT8、INT16、FP16三种数据格式，支持流水线无缝切换指令。

#### PE_RCTL（PE读控制器）

负责从L2中读取数据，分发给16个PE的activation输入。

支持4种数据分发模式：
1. 全部16个PE用相同的activation数据（广播）；
2. 4组PE，组间数据相同，组内PE数据不同；
3. 4组PE，组间数据不同，组内PE数据相同；
4. 16个PE各用不同的数据。

支持正常卷积、反卷积、组卷积，支持四个方向不对称padding，支持w/h方向不同stride（步长）和不同kernel size，范围均支持到1~32。

#### PE_WCTL（PE写控制器）

负责接收16个PE的输出数据，写回L2 Buffer。

支持**增量写操作**（先读出L2中已有的数据，做反量化，与PE输出的部分和相加，再写回——这对于多PE切分输入通道协同计算很关键）。支持激活函数和量化操作的可选配置。

#### PE（处理单元）

每个PE包含：Router（路由器）、AL1 Buffer、WL1 Buffer、8-lane ALU（**ALU，Arithmetic Logic Unit，算术逻辑单元，执行实际乘加运算的部件**）、BN模块。每个ALU包含8个MAC（**MAC，Multiply-Accumulate，乘累加运算单元**）。

**Router（路由器）：** 决定输入的activation/weight数据是留在本地PE使用还是继续转发给下一个PE（通过数据id选择）。

**WL1（权重缓存）：** 8组，每组接受对应id的weight数据，给8个MAC提供权重。

**AL1（激活缓存）：** 接收activation数据并输出给MAC，支持ping/pong双缓冲和跨指令数据复用。

**ALU（乘累加单元）：** 支持的计算模式有：
- INT计算：4bit×4bit、8bit×4bit、8bit×8bit、16bit×8bit
- 浮点计算：BF16×BF16

**BN模块：** 负责执行批归一化（BN）和激活函数，支持定点转浮点（FP32乘加运算）和浮点转定点，支持所有算子表中的激活函数。

#### Vector Processor（向量处理器）

负责执行所有非卷积类的"后处理"算子：Pooling（池化）、Element-wise Mul（逐元素乘法）、上采样等。支持4路64bit读写端口，支持流水线操作。

### 算子应用方案（4.5节末尾）

列出了10种典型的算子融合链路，从简单到复杂：

1. FP16 conv（单纯卷积）
2. FP16 conv + BN（卷积+批归一化）
3. FP16 conv + BN + 激活函数
4. FP16 conv + BN + 激活 + 量化
5. FP16 conv + BN + add + 激活 + 量化（包含残差相加，如ResNet的skip connection）
6. uint8/uint16 conv + 反量化
7. uint8/uint16 conv + 反量化 + BN
8. uint8/uint16 conv + 反量化 + BN + 激活函数
9. uint8/uint16 conv + 反量化 + BN + 激活 + 量化
10. uint8/uint16 conv + 反量化 + BN + add + 激活 + 量化

这些融合方案的意义在于：把多个独立的操作步骤合并成一次硬件执行，避免中间数据反复在片上缓存和片外内存之间来回搬运，大幅提升效率。
